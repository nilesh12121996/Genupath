Layer 0 · Security baseline — always-on infrastructure

HashiCorp Vault persistent service

Dynamic short-lived tokens · PII encryption at rest · mTLS between all services · secrets rotation policy

  • Never in Agent 0 setup — always-on sidecar every agent calls at runtime
  • Each agent gets a scoped token valid for its run only (least-privilege)
  • PII fields (name, address, EEO answers) AES-256 encrypted at rest in PostgreSQL
  • mTLS between all service-to-service calls — no plain HTTP internally

GDPR / compliance layer new

Data retention policy · right-to-delete pipeline · ToS version monitor · application audit log

  • PostgreSQL row-level TTL: job records auto-purge after 12 months
  • Full audit log of every action taken on your behalf (immutable append-only)
  • ToS change monitor: weekly diff of job board ToS pages — Slack alert on change
  • EEO answers flagged as sensitive — excluded from all LLM context windows

Deployment layer new

Docker Compose (dev) · Kubernetes (prod) · GitHub Actions CI/CD · dev / staging / prod envs

  • Each agent is a containerized Python service — independent deploy and rollback
  • Prompt changes to Agent 3 go through staging with ATS score regression test before prod
  • Feature flags (LaunchDarkly or simple DB flags) for gradual agent rollout
  • Resource sizing: Playwright worker nodes need 2 vCPU / 4GB RAM each

Layer 1 · Event trigger layer — push-only, zero polling

Gmail Pub/Sub webhook

LinkedIn + Indeed job alerts → dedicated Gmail → Pub/Sub push → event queue within seconds

  • Dedicated non-personal Gmail address for all job alerts
  • Gmail API watch() registers a Pub/Sub subscription on the inbox
  • New email arrives → Pub/Sub push → Lambda → parse URL + title → enqueue

WebSub / RSS feeds

Superfeedr aggregator · Indeed Publisher API · Greenhouse job board RSS · push on new item

  • No polling — Superfeedr calls your webhook when a new item appears in any subscribed feed
  • Covers RemoteOK, Lever, Ashby, Greenhouse board feeds

DOM hash watcher + rate limiter

Headless diff on target company /jobs pages · per-platform daily caps enforced at entry

  • Hash watcher fires only on DOM change — not on schedule
  • Rate limiter at event ingress: LinkedIn 15/day · Indeed 25/day · direct company pages 50/day
  • Circuit breaker: if a platform 429s 3× in a row, pause that source for 2 hours

Vector dedup gate new

JD title + description embedded → cosine similarity check → drops near-duplicate postings before processing

  • Embed incoming JD title + first 300 chars into Pinecone at ingestion
  • If cosine similarity > 0.92 to any job processed in last 30 days → drop as duplicate
  • Prevents processing the same role reposted under a slightly different title

Layer 2 · Agent 0 — one-time intake & profile bootstrap

Resume & experience ingestion

Claude Sonnet parses PDF → PostgreSQL structured tables → Pinecone embeddings · versioned

  • Extracts: roles, dates, bullet points, tech stack, quantified achievements separately
  • Version-controlled: each resume update creates a new version_id, old versions preserved
  • Truth constraint seeded here: list of every company, role, and date range you've actually held

Browser profile factory new

Creates aged browser profiles with history, cookies, saved passwords — looks like a real 6-month-old account

  • Each portal gets its own Playwright persistent context stored on disk
  • Profile "aging": pre-populate browsing history, LinkedIn activity, Indeed searches before first real run
  • Warm-up phase: 2 weeks of light browsing (job searches, profile views) before first application
  • Profiles stored encrypted in object storage — reused per session, never recreated fresh

Market intelligence baseline new

Levels.fyi + Glassdoor scrape for target roles · sets salary expectation floor · company health signals

  • One-time scrape of compensation data for your target title + location
  • Seeds a salary_expectations table: P25 / median / P75 by company and role
  • Agent 3 uses this to populate salary fields realistically — never leaving them blank

Layer 3 · Phase 1 — discovery & validation (sequential)

Agent 1 · Scout

Event-driven Playwright · HTML → JSON · GPT-4o-mini · schema version registry

  • Schema version registry: each job board has a versioned CSS selector config — updated independently of agent code
  • If extraction fails on >3 consecutive jobs from one source → flag that source for selector update, not a full agent failure
  • Outputs canonical JD JSON: title, company, location, remote_ok, skills[], yoe_required, salary_range, ats_platform

Agent 2 · Intel

DOL OFLC H-1B disclosure data · myvisajobs.com API · Glassdoor + Crunchbase enrichment

  • Source: DOL OFLC H-1B Disclosure Data (flcdatacenter.com) — the real public dataset. "USCIS Torch API" does not exist.
  • Probability score = (LCA filings in last 2 years) weighted by recency + Glassdoor sponsorship reviews
  • Also pulls Crunchbase: funding stage + runway signal — avoids applying to companies likely to do layoffs
  • Gate: H1B score < 0.7 → halt · funding_status = "acquired" or "shutdown" → halt

ATS platform detector new

Identifies Workday / Greenhouse / Lever / Taleo / Ashby / direct — routes to correct submission adapter

  • Regex + HTTP header fingerprinting on the apply URL to classify the ATS platform
  • Each platform has its own submission adapter with known field mappings
  • Workday is the hardest — has its own state machine. Taleo is the most brittle — needs special handling.
  • Unknown ATS → Computer Use fallback (slower but generic)

Layer 4 · Phase 2 — generation (runs after Phase 1 gate passes)

Agent 3 · Architect

Claude Sonnet · semantic bullet retrieval · truthfulness gate · LaTeX PDF · ATS loop (3× hard cap)

  • Semantic retrieval: embed JD requirements → find top-k matching experience bullets in Pinecone
  • Truthfulness gate: LLM output cross-checked against truth_constraints table — any company/date/role not in your real history is rejected and rewritten
  • ATS loop capped at 3 iterations — fail → "manual review" alert, never infinite
  • Resume version_id stored with every application for outcome tracking

Cover letter + salary branch

Detects requirement from JD schema · Claude Sonnet letter · salary expectation auto-populated from market data

  • Cover letter: 3 paragraphs — hook (specific company insight), match (your strongest 2 aligned achievements), close
  • Salary field: auto-populated from salary_expectations table at P60 (negotiable but realistic)
  • If JD has no salary range listed → leave field blank unless portal requires it

Jobscan ATS scorer new

Jobscan API replaces mock-ATS — real keyword match scoring against actual ATS algorithms

  • Jobscan has an API that scores resumes against JDs using real ATS matching logic
  • More reliable signal than a home-built mock parser
  • Score + missing keywords returned → Agent 3 uses missing keywords to guide rewrites
  • Target: ≥ 80% Jobscan match (90% was unrealistically high for most real JDs)

Layer 5 · Human behavior engine — cross-cutting service, injected into Agents 1, 4, 5

Fingerprint & network layer

playwright-extra stealth · Brightdata residential proxy (sticky per domain) · canvas + WebGL + AudioContext randomization

  • playwright-stealth masks: navigator.webdriver, chrome.runtime, plugins array, permissions API
  • Also mask: WebGL renderer string, AudioContext fingerprint, screen color depth
  • Brightdata sticky session: same residential IP for entire session per domain
  • TLS fingerprint (JA3) randomization via custom ClientHello — defeats Cloudflare/Datadome at the TCP layer

Behavioral timing engine

Gaussian delays · reading simulation · business-hours gate · platform-specific behavior profiles

  • Timing drawn from Gaussian (μ=1.4s, σ=0.5s) — never fixed delays
  • Business hours gate: 8am–7pm local · no submissions on weekends (bot signal)
  • Platform profiles: LinkedIn has slower, more deliberate scrolling than Indeed
  • Rage-click prevention: never click same element twice in under 800ms

Mouse, keyboard & session

Bezier paths · 45–65 WPM with typos · session warm-up browse · persistent aged profile per domain

  • Mouse: Bezier curve to target, slight overshoot + correction (~20% of moves)
  • Keyboard: per-character timing variance, 1-in-40 typo + backspace
  • Session warm-up: visit homepage → scroll → click 2 nav items → wait 45–90s → navigate to jobs
  • Persistent profile context reused — session history, cookies, local storage all intact from last visit

CAPTCHA + detection response

2captcha primary · CapSolver fallback · Datadome/Cloudflare bypass layer · human-in-loop Slack alert

  • hCaptcha / reCAPTCHA v2: 2captcha API (solve time ~15s)
  • reCAPTCHA v3 (score-based): requires behavioral warm-up before it triggers — handled by session aging
  • Datadome / Cloudflare Bot Management: defeated by TLS fingerprint randomization + residential IP
  • If blocked after 2 retries → pause agent + Slack alert with screenshot → await manual clearance
↓

Layer 6 · Phase 3 — submission & networking

Agent 4 · Pilot

Skyvern primary (ATS-aware) · Computer Use fallback · Browserbase sessions · saga pattern for rollback

  • Skyvern as primary: it has a hosted API, ATS-specific task templates, and handles Workday/Greenhouse natively — more reliable than raw Computer Use on complex forms
  • Computer Use as fallback only for unknown ATS types
  • Saga pattern: each form step checkpointed — if Agent 4 crashes mid-form, next run resumes from last checkpoint, not from scratch
  • Idempotency key per application: prevents double-submit on retry

Agent 5 · Networker

Proxycurl API primary · throwaway account fallback · Claude Sonnet DM · hiring manager vs recruiter classifier

  • Proxycurl API: legitimate paid LinkedIn data API that doesn't require scraping (~$0.01/profile). No ToS risk. Replaces impossible "LinkedIn People Search API".
  • Hiring manager classifier: GPT-4o-mini checks job title to prioritize engineering manager or director over generic recruiters
  • DM sent with 3–6 min reading delay · references one specific, genuine company detail (recent news, product, blog post)
  • Throwaway account fallback: rate-capped at 8 profile views/hour, never personal account
↓

Layer 7 · Phase 4 — follow-up, learning & interview prep

Agent 6 · Follow-up

5-day DM follow-up · Pinecone context recall · max 1 per application · interview invite detector

  • Checks email inbox for interview invites — if found, cancels follow-up and fires Agent 7
  • Follow-up DM references original message naturally using Pinecone recall
  • Max 1 follow-up. If no reply → status = "closed", no further outreach

Outcome A/B framework new

Tracks resume_version → interview rate · statistical significance test before strategy switch

  • PostgreSQL outcomes table: job_id, resume_version_id, ats_score, jd_similarity, status, response_days
  • After N=30 applications: Chi-square test on interview rate per resume strategy
  • Only switches Agent 3 strategy when p < 0.05 — prevents noise-driven changes

Agent 7 · Interview prep new

Triggered on interview invite · JD-tailored STAR questions · Pinecone Q&A coaching · company deep-dive brief

  • Triggered when Agent 6 detects an interview invite in the Gmail inbox
  • Generates: 10 role-specific behavioral questions (STAR format), 5 technical questions from JD stack, company background brief (Glassdoor culture, recent news, leadership team)
  • Answers drawn from your Pinecone interview memory — personalized, not generic
  • Delivered as a formatted PDF + Slack notification

Layer 8 · Orchestration — the nervous system

LangGraph + LangSmith

State machine · conditional edges · retry with exponential backoff + jitter · full trace per run

  • Exponential backoff: base=2s, max=5 retries, ±20% jitter — avoids thundering herd on retries
  • Circuit breaker per external API: trips after 5 consecutive failures, resets after 10 min
  • LangSmith traces every node: input hash, output hash, token count, latency — full audit trail

Redis Streams event bus

Async agent comms · consumer groups · dead-letter queue · event replay for debugging

  • CloudEvents spec for event schema: source, type, datacontenttype, time, id fields standardized
  • Event versioning: schema_version field on every event — consumers handle v1 and v2 simultaneously during rollout
  • Dead-letter queue: events that fail 3× processing → DLQ → Slack alert for manual inspection
  • Event replay: any stream can be replayed from offset for debugging or backfill

Rate limiter service new

Token bucket per platform · global daily application cap · SMS/push on interview invite

  • Token bucket algorithm: refills at platform-specific rate — burst-safe, not just daily cap
  • Global hard cap: max 20 applications/day total (above this = obvious bot signal)
  • Interview invite detector: monitors Gmail for subject-line patterns → fires immediate SMS + Slack push
↓

Layer 9 · Data layer

PostgreSQL

Jobs · experiences · outcomes · versions · audit log · salary benchmarks · truth constraints

  • Row-level security: EEO/diversity fields isolated to a separate encrypted table, never joined into LLM context
  • Audit log table: append-only, no UPDATE/DELETE — full immutable history of every agent action
  • Automatic TTL: job records purge after 12 months (GDPR compliance)

Pinecone + Redis + Object storage

Vector memory (interview Q&A, JD embeddings) · session cache · resume PDFs · browser profiles

  • Pinecone namespaces: interview-memory, jd-archive, cover-letter-bank
  • Redis: session state cache, rate limiter token buckets, dedup bloom filter
  • Object storage (S3/R2): versioned resume PDFs, cover letters, browser profile snapshots

Observability stack

LangSmith traces · Datadog metrics · Slack alerts · token budget guardrail · SLA dashboard

  • Daily token budget: decremented per API call, auto-pause all agents if breached
  • SLA targets: event-to-application < 8 min · DM sent < 12 min · interview prep delivered < 5 min of invite detection
  • Weekly digest: applications sent, interview rate %, token spend, blocked-by-CAPTCHA count

1

Secrets Vault placed inside Agent 0 (fault)

Vault shown as a one-time setup task in Agent 0 means it only exists at init. Every agent needs live secrets at runtime. Vault must be always-on infrastructure that every agent calls independently via scoped tokens.
2

No truthfulness gate on resume rewriter (fault)

Claude Sonnet rewrites bullets without a constraint on what's real. Without a truth_constraints table (your actual companies, roles, dates), the LLM can plausibly hallucinate a bullet. That is resume fraud. The gate is non-negotiable.
3

No saga / checkpoint on Agent 4 (fault)

Workday and Greenhouse forms have 4–8 sequential pages. If Agent 4 crashes on page 5, the next run restarts from page 1 and double-submits or corrupts the application. Checkpoint-per-page and an idempotency key are essential.
4

No ATS platform detection or adapters (fault)

Workday, Greenhouse, Lever, Taleo, and Ashby all have completely different DOM structures. A single generic Computer Use agent will fail on ~60% of applications. Each major platform needs its own field-mapping adapter.
5

No deployment layer or environment separation (fault)

Without dev/staging/prod environments, every prompt change to Agent 3 goes straight to production, potentially breaking ATS scores for live applications. CI/CD with staging regression tests before any prod deploy is required.
6

90% ATS score target is unrealistically high (caution)

Home-built mock-ATS parsers rarely correlate with real ATS behavior. Replaced with Jobscan API (real ATS matching algorithms) and an 80% target — more achievable and better calibrated to what actually gets you through screening.
7

Browser profiles created fresh each run (caution)

A new browser profile with zero history is one of the strongest bot signals. Real users have months of browsing history, saved passwords, and cookies. Profile aging (2-week warm-up, persistent storage per domain) is what actually defeats modern fingerprinting.
8

No statistical rigor on A/B outcome feedback (caution)

Switching resume strategies based on a few applications is noise-driven, not signal-driven. The outcome framework now only triggers a strategy switch when Chi-square p < 0.05 across at least 30 samples per variant.
9

Added vector dedup gate at event ingress (improvement)

The same role reposted weekly (common on Indeed) would trigger the full pipeline every time without this. Cosine similarity > 0.92 on embedded JD title + description catches reposts before any expensive processing begins.
10

Added Agent 7: Interview prep agent (improvement)

The system currently ends at follow-up. When an interview is booked, Agent 7 auto-generates JD-tailored STAR questions, technical prep, and a company brief using the same Pinecone memory already in the stack. Closes the full job-search lifecycle.
11

Added TLS fingerprint randomization (improvement)

playwright-stealth handles browser-level JS API masking but doesn't touch the TLS handshake (JA3 fingerprint). Cloudflare Bot Management and Datadome fingerprint at the TCP layer. Custom ClientHello configuration defeats this class of detection entirely.
12

Added market intelligence + salary auto-population (improvement)

Leaving salary fields blank or writing "negotiable" is a weak move. Pre-seeding salary data from Levels.fyi/Glassdoor and auto-populating at P60 for your role makes every application more complete and avoids screener rejection for skipped required fields.

!

"USCIS Torch API" does not exist

There is no API called "USCIS Torch API." USCIS does not offer a developer API for LCA data. The real source is the DOL Office of Foreign Labor Certification (OFLC) H-1B Disclosure Data at flcdatacenter.com — a downloadable public dataset updated quarterly. Use myvisajobs.com or h1bgrader.com as secondary lookup APIs.
!

LinkedIn People Search API is not publicly available

LinkedIn's People Search API requires LinkedIn Partner Program approval and is restricted to enterprise partners. It is not available to individual developers or small teams. The correct replacement is Proxycurl — a legitimate paid API that legally aggregates LinkedIn public profile data at ~$0.01/profile with no ToS risk.
!

playwright-stealth alone does not defeat Cloudflare

playwright-stealth patches browser JS APIs (navigator.webdriver etc.) but does nothing about the TLS handshake (JA3 fingerprint), HTTP/2 frame ordering, or behavioral mouse entropy that Cloudflare Bot Management and Datadome use. You also need TLS fingerprint randomization and residential IPs to defeat these at the network layer.
~

Skyvern is real but primarily a hosted service

Skyvern exists and has a hosted API that handles Workday, Greenhouse, and other ATS forms well. However it is not free and the self-hosted open-source version is significantly less capable than the hosted version. Budget ~$0.10–0.50 per form submission. Use the hosted API, not self-hosted.
~

Anthropic Computer Use is real but unreliable on complex forms

Anthropic's Computer Use API works well for simple forms but has a ~30–40% failure rate on multi-page ATS systems like Workday (which uses a custom SPA framework). It should be a fallback for unknown ATS types, not the primary submission mechanism. Skyvern with ATS-specific templates is more reliable as primary.
~

Browserbase is real — use it correctly

Browserbase is a real headless browser cloud platform. It does provide some bot-detection evasion but its IP pool is data-center based, not residential — meaning LinkedIn and advanced portals will still flag it without a separate residential proxy layer. Use Browserbase for session infrastructure, route traffic through Brightdata residential IPs.
~

Gmail Pub/Sub is real but requires domain verification

Gmail's Pub/Sub push notification API (via Google Cloud Pub/Sub) requires you to verify domain ownership of the email address or configure a service account. For a Gmail.com address (not G Suite), it works but the watch() subscription expires every 7 days and must be renewed — build an auto-renewal Lambda/cron for this.
+

Proxycurl — the realistic LinkedIn alternative

Proxycurl (proxycurl.com) is a paid API that provides LinkedIn profile data without scraping. Pricing is ~$0.01/profile lookup. It covers the recruiter/hiring manager discovery use case Agent 5 needs, works within legal bounds, and has no risk of account bans. It is the correct realistic replacement for the unavailable People Search API.
+

Jobscan API — the realistic ATS scorer

Jobscan has a developer API that scores resumes against job descriptions using real keyword matching algorithms that mirror actual ATS behavior. It returns a match percentage plus a list of missing keywords — directly actionable for Agent 3's rewrite loop. Far more reliable than a home-built parser.
+

DOL OFLC data — the real H-1B source

The Department of Labor publishes H-1B LCA disclosure data quarterly at flcdatacenter.com. It includes employer name, job title, wage, work location, and approval status. This is the actual dataset used by myvisajobs.com and H1BGrader. Download it, load into PostgreSQL, and query locally — faster and more reliable than any third-party API wrapper.
